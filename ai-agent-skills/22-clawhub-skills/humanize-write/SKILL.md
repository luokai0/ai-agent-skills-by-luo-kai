# 《说人话》

> 触发词：「说人话」「去AI味」「润色文案」「生成正文」「human写作」「写文章」
> 适用场景：热点配图完成后，进入正文创作阶段
> 平台适配：小红书 / 抖音 / 微信公众号 / 微博
> 版本：1.2.0
> ⚠️ 核心原则：
> - 本技能面向通用用户，不预设称呼
> - 用户文风偏好统一存入 `/workspace/memory/evolution/user-writing-style.md`

---

## 创作全流程（5步）

```
Step1 确认时间锚点（询问用户）
Step2 搜索关联背景（3-6个月）+ 判断时效性
Step3 确认平台
Step4 确认创作视角/人称
Step5 结合平台特征产出内容
```

---

## Step1：确认时间锚点

**⚠️ 必须询问用户，不能自行决定。**

### 询问话术

```
需要确认一下时间范围：
1）用今天作为锚点，自动推算近3-6个月
2）我来指定一个时间范围（请告诉我具体时间）
请选择：1 或 2
```

### 选项1：自动锚点（联网确认）

1. 联网查询今天的准确日期
   - 调用 UAPI `/misc/worldtime` 接口
   - 或用搜索API查询「今天日期 2026」
2. 记录锚点：今天是星期X，YYYY年MM月DD日
3. 自动推算：
   ```
   近3个月：YYYY年(M-3)月 ~ YYYY年MM月
   近6个月：YYYY年(M-6)月 ~ YYYY年MM月
   近1年：YYYY年(M-12)月 ~ YYYY年MM月
   ```

### 选项2：用户指定时间

用户直接说出时间范围，例如：
- 「就用2025年10月到2026年3月」
- 「用去年全年的数据」

收到后直接使用，不做校验。

---

## Step2：搜索关联背景 + 判断时效性

### 搜索策略

围绕热点主题，在网上搜索Step1确认的时间范围内的相关内容：

**搜索维度：**
```
1. 直接相关新闻（同一事件的早期报道）
2. 行业背景（导致这件事的上下游原因）
3. 相关数据（同期市场数据、对比数字）
4. 行业人士/专家解读（观点而非新闻）
5. 普通人反应（论坛、评论区真实声音）
```

### 判断时效性

搜索完成后，判断这个热点：

```
✅ 高度时效：今天/昨天发生的事件 → 快发，抢第一波流量
⏰ 中等时效：事件已有2-7天 → 补充独特视角，不拼速度
📌 弱时效：事件已有1个月以上 → 做深度分析/盘点类内容
❌ 过期热点：超过1个月 → 不做，或转盘点类
```

### 输出格式

```
📊 背景搜索结果：

事件关联（近X个月）：
  → [找到的相关背景1]
  → [找到的相关背景2]
  → [找到的相关背景3]

时效性判断：[高度/中等/弱/过期]
核心原因假设：[2-3个最可能的深层原因]
内容角度建议：[从哪个角度切入最有话说]
```

## Step3：确认发布平台

向用户确认：

```
这个话题适合做[深度分析/快讯/盘点/观点文]类型的内容。
请问发布在哪个平台？
  1）小红书
  2）抖音
  3）微信公众号
  4）微博
```

---

## Step4：确认创作视角/人称

**⚠️ 必须询问用户，不能自行决定。**

### 询问话术

```
创作视角想怎么定？
  1）我帮你选一个合适的（默认：我朋友/路人/从业者等）
  2）你来说每次想要什么人称
  3）固定用某种人称（请告诉我是什么，比如"作为一个普通消费者"）
```

### 三种选择

| 选项 | 说明 | 示例 |
|------|------|------|
| 1 | 默认AI推荐 | 「作为一个手机爱好者」「消费者视角」 |
| 2 | 用户每次指定 | 本次用「从业者」，下次用「消费者」 |
| 3 | 固定人称 | 始终用「消费者视角」，不用再问 |

**注意：** 选择选项3的用户，其固定人称偏好也存入 `user-writing-style.md`。

### 各平台内容特征

**小红书：**
- 结构：开头钩子 → 个人经历/感受 → 2-3个观点 → 结尾互动
- 语气：日常分享感，像朋友聊天，微微吐槽
- 段落：短段落，每段不超过3行
- 字数：300-800字
- 结尾：提问/引导评论/留悬念

**抖音：**
- 结构：开场3秒钩子 → 第一段观点（20秒）→ 第二段细节（30秒）→ 金句（15秒）→ 互动引导
- 语气：有态度，敢说，有点偏激更好
- 字数：200-400字口播稿

**微信公众号：**
- 结构：开门见山 → 2-3个论点 → 总结升华
- 语气：相对正式，但口语化，不学究
- 字数：800-2000字
- 结尾：引导关注/点在看

**微博：**
- 结构：1-2句话 + 观点 + 标签
- 语气：简短有力，有态度
- 字数：100-300字

## 写作联想层（写文前的思维过程）

写正文之前，先跑一遍这个思维流程：

### Step 0a：判断相关性

拿到热点主题，先问自己：

- **这个事件的核心原因是什么？** （往下挖一层，不只看表面）
- **除了新闻里写的，还有哪些可能的原因？** （大胆猜测，有据可依）
- **我作为一个普通人，怎么理解这件事？** （真实的人，不是分析师）

### Step 0b：联想行业上下文

围绕主题，强制联想3个相关维度：

```
例子：笔记本电脑暴跌40%（2026年1-2月数据）
→ 直接原因：消费降级、替换周期延长
→ 隐藏原因：内存/SSD在2024-2025年涨价，整机价格上去了
→ 替代品：手机算力提升、轻办公平板流行
→ 从业者视角：OEM品牌商、渠道商压力
→ 用户行为：能用手机就不开电脑
→ 补充角度：RTX40系列显卡换代周期延长，游戏本需求也在降
```

### Step 0c：选择最有话说2-3个点

- 不是所有联想都写进文章
- 选**最有把握、最有话说**的2-3个点
- 每个点要有**一句有观点的判断句**（不是疑问句，是肯定句）
- 猜测要有逻辑：「我觉得……因为……」

### Step 0d：把判断写进正文

- 不是补充资料，是**融入个人判断**
- 用「我觉得」「说白了」「问题是」等个人口吻
- 不要列数据，要讲逻辑：「电脑卖不动，不只是因为贵，是因为连'为什么要买'都说不清了」

## 去AI味6步操作清单

### Step 1：打乱结构

删除所有"首先/其次/第三/最后/综上所述"，把完整长句拆成短句。

**改前：** `首先，我们需要了解AI的基本原理，其次，要掌握提示词技巧，最后才能生成高质量内容。`  
**改后：** `AI这东西，说白了就是喂多少料出多少货。你不用懂技术，但得学会怎么跟它说话。`

### Step 2：口语化替换

| 书面词 | 口语替换 |
|--------|---------|
| 首先 | 其实 / 说实话 / 第一步 |
| 其次 | 而且 / 接着 / 然后 |
| 综上所述 | 总之 / 说白了 / 重点是 |
| 值得注意的是 | 关键来了 / 划重点 |
| 据悉 | 听说 / 好像 |
| 非常重要 | 真的挺关键的 / 千万别忽略 |

### Step 3：加语气词和情绪

在句首加语气词：「说实话」「其实」「我觉得」「你别说」「真的」  
在情绪点加感叹：「也太香了吧」「这谁顶得住」「救命🆘」

### Step 4：加入个人视角

强制添加"我"的经历或感受：

**改前：** `AI可以显著提升写作效率。`  
**改后：** `我之前写一篇内容要2小时，用了AI之后，半小时搞定，真的香。`

### Step 5：破坏均匀节奏

- 至少一段只有1-2句话（制造"想到哪说到哪"感）
- 至少一段超过6句话（制造信息密度感）
- 偶有不完整句，句号用在一半就停了
- 感叹号和问号混用

### Step 6：删掉升华结尾

**改前：** `以上就是关于AI写作的全部内容，希望对您有所帮助，感谢观看。`  
**改后：** `反正我是回不去了。你们自己看着办吧。`

## 用户写作风格记录机制

### 核心原则

- 称呼通用用户为「你」或直接用户名，不预设特殊称呼
- 用户写作风格偏好存入：`/workspace/memory/evolution/user-writing-style.md`
- 每次写正文前，读取该文件作为参考基准

### 文风文件路径规范

```
用户文风记录：/workspace/memory/evolution/user-writing-style.md
  - 记录用户的写作偏好、禁忌、口头禅、改稿规律
  - 由技能执行者主动维护，每次用户改稿后更新

进化日志：/workspace/memory/evolution/YYYY-MM.md
  - 每次技能更新产生的变化记录
  - 包含操作类型、原因、预期效果
```

### 用户文风记录触发时机

每次用户修改正文后，主动执行以下操作：

1. **识别**：从用户修改版本中提取文风特征（开头方式/语气/段落节奏/结尾模式）
2. **对比**：与当前记录差异，如有无新增偏好或禁忌
3. **更新**：将差异部分追加写入 `user-writing-style.md`

### 更新格式

```
## [YYYY-MM-DD HH:MM] 文风更新

- **新增特征**：[用户的独特写法]
- **触发场景**：[什么类型的热点/平台]
- **原写法**：[之前怎么写]
- **新写法**：[用户改成什么样]
```

## 注意事项

- **时间锚点必须询问**：不能自行决定用哪个时间
- **视角/人称必须询问**：不能自行代入特定人称
- **文风文件路径必须正确**：错误路径会导致文风丢失
- **每次正文输出前必须读取** `user-writing-style.md`
- 所有用户统一称「你」或直接用户名，不预设特定称呼
